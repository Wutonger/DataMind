# 博客系统专业说明书

## 1. 文档说明

本文档用于描述博客系统的业务目标、系统架构、核心数据模型、关键流程、接口约定、报表指标与运行规范。  
文档适合作为知识库测试样例，可用于验证以下能力：

- 文档上传与解析
- 分块检索与引用定位
- 基于文档的问答
- 文档与数据库联合分析

本文档中的业务规则、字段定义、流程约束和术语口径，均以本说明书为准。

---

## 2. 系统概述

博客系统是一套面向内容创作、发布与运营分析的内容管理平台，主要服务于个人博主、小型内容团队与企业内容运营部门。  
系统目标包括：

- 支持文章创作、审核、发布、下线的全生命周期管理
- 支持标签、分类、作者、评论、附件等内容关联能力
- 支持按时间、作者、分类、标签维度进行数据统计
- 支持对外提供文章列表、详情、搜索与推荐接口
- 支持后台进行内容运营、用户管理与数据分析

博客系统由前台展示端、后台管理端、内容服务层、检索与推荐组件、数据分析模块组成。  
在部署模式上，系统通常分为开发环境、测试环境和生产环境，其中生产环境要求启用日志审计、访问限流和数据备份。

---

## 3. 业务目标

### 3.1 内容生产目标

内容生产目标是提升文章产出效率、保证内容质量并缩短发布时间。系统要求编辑在后台可完成草稿保存、版本回溯、富文本编辑、封面管理和定时发布。

### 3.2 内容运营目标

内容运营目标是提升文章曝光、阅读深度与用户互动。系统需支持热门文章榜单、标签聚合页、分类页、站内搜索、相关推荐与评论互动。

### 3.3 数据分析目标

数据分析目标是帮助运营和管理人员回答以下问题：

- 哪些文章阅读量最高
- 哪些分类表现最好
- 哪位作者产出最多
- 用户最关注哪些标签
- 评论与阅读量之间是否存在明显相关性
- 最近 30 天内容发布节奏是否稳定

## 4. 角色与权限

系统内置四类主要角色：访客、作者、编辑、管理员。

### 4.1 访客

访客可以浏览已发布文章、按分类或标签筛选文章、查看评论、进行站内搜索。  
访客默认没有后台权限，也不能查看未发布内容。

### 4.2 作者

作者可以创建草稿、编辑自己名下的文章、提交审核、查看自己的内容统计。  
作者不能直接删除已发布文章，也不能修改其他作者内容。

### 4.3 编辑

编辑可以审核文章、修改分类标签、安排发布时间、下线文章。  
编辑对内容质量、合规性和页面展示负责。

### 4.4 管理员

管理员具有全局权限，包括用户管理、角色分配、系统配置、数据回收、报表查看与导出。  
只有管理员可以修改敏感配置，如评论审核策略、上传大小限制、搜索热词配置和推荐算法开关。

### 4.5 权限原则

权限控制遵循最小权限原则。  
作者只能访问自己的文章数据；编辑只能操作内容相关数据；管理员可访问全局数据和系统配置。  
删除操作通常采用逻辑删除，不允许普通角色执行物理删除。

## 5. 核心业务对象

博客系统主要包含以下业务对象：

- 用户 `user`
- 作者档案 `author_profile`
- 文章 `post`
- 分类 `category`
- 标签 `tag`
- 文章标签关系 `post_tag`
- 评论 `comment`
- 附件 `attachment`
- 浏览记录 `post_view_daily`

### 5.1 用户

用户是系统中的身份主体。  
关键字段包括：

- `id`：用户主键
- `username`：登录名，唯一
- `display_name`：展示名称
- `email`：邮箱
- `role`：角色，枚举值为 `visitor`、`author`、`editor`、`admin`
- `status`：状态，枚举值为 `active`、`disabled`
- `created_at`：创建时间

### 5.2 作者档案

作者档案用于扩展用户信息，记录作者简介、头像、社交链接和签名。  
一个作者档案只对应一个用户。

### 5.3 文章

文章是系统核心对象。  
关键字段包括：

- `id`：文章主键
- `title`：文章标题
- `slug`：URL 别名，唯一
- `summary`：摘要
- `content_markdown`：Markdown 正文
- `content_html`：渲染后的 HTML
- `author_id`：作者 ID
- `category_id`：分类 ID
- `status`：状态
- `is_top`：是否置顶
- `allow_comment`：是否允许评论
- `published_at`：发布时间
- `created_at`：创建时间
- `updated_at`：更新时间

### 5.4 文章状态定义

文章状态枚举值说明如下：

- `draft`：草稿
- `reviewing`：审核中
- `scheduled`：定时发布
- `published`：已发布
- `archived`：已归档
- `offline`：已下线

其中，只有 `published` 状态的文章会出现在前台列表和搜索结果中。

### 5.5 分类

分类用于表达文章主题归属，例如“后端开发”“数据库”“产品思考”“运营实践”。  
一个文章只能属于一个主分类。

### 5.6 标签

标签用于表达文章的细粒度主题，例如“Spring Boot”“MySQL”“缓存”“推荐系统”。  
一个文章可以绑定多个标签，因此文章与标签是多对多关系。

### 5.7 评论

评论用于记录用户互动。  
评论字段包括：

- `post_id`
- `nickname`
- `content`
- `status`
- `reply_to_comment_id`
- `created_at`

评论状态通常包括：

- `pending`
- `approved`
- `rejected`
- `deleted`

## 6. 关键业务流程

### 6.1 文章创建流程

文章创建流程如下：

1. 作者进入后台创建新文章
2. 输入标题、摘要、正文、分类和标签
3. 系统自动保存草稿
4. 作者提交审核
5. 编辑审核通过后发布，或驳回后退回草稿

### 6.2 审核发布流程

审核流程要求编辑检查以下内容：

- 标题是否准确
- 内容是否完整
- 是否存在错别字或敏感信息
- 分类与标签是否合理
- 封面图与摘要是否合规

审核通过后，文章可立即发布，也可设置定时发布时间。  
定时发布时间必须晚于当前系统时间至少 5 分钟。

### 6.3 评论审核流程

若系统启用“评论先审后发”，新评论默认进入 `pending` 状态。  
编辑或管理员审核通过后，评论进入 `approved` 状态并显示在前台。  
若评论包含广告、辱骂或敏感词，应标记为 `rejected`。

### 6.4 内容下线流程

当文章存在政策风险、版权争议或信息过时问题时，可执行下线操作。  
下线后文章状态变为 `offline`，前台不再展示，但后台仍保留记录。  
只有管理员和编辑可以执行下线。

## 7. 数据库设计说明

### 7.1 主要数据表

系统核心表如下：

- `users`
- `author_profiles`
- `posts`
- `categories`
- `tags`
- `post_tags`
- `comments`
- `attachments`
- `post_view_daily`

### 7.2 posts 表说明

`posts` 表是博客系统最重要的数据表。  
推荐字段如下：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 主键 |
| title | varchar(255) | 标题 |
| slug | varchar(255) | URL 别名 |
| summary | text | 摘要 |
| content_markdown | longtext | Markdown 正文 |
| content_html | longtext | HTML 正文 |
| author_id | bigint | 作者 ID |
| category_id | bigint | 分类 ID |
| status | varchar(32) | 文章状态 |
| is_top | tinyint | 是否置顶 |
| allow_comment | tinyint | 是否允许评论 |
| published_at | datetime | 发布时间 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 7.3 comments 表说明

`comments` 表用于存储评论内容。  
一篇文章可以有多条评论，一条评论也可以回复另一条评论。  
因此 `reply_to_comment_id` 可为空，为空表示一级评论，不为空表示回复评论。

### 7.4 post_view_daily 表说明

`post_view_daily` 表用于存储文章按天聚合的浏览统计。  
该表通常由埋点或日志汇总任务生成。  
关键字段包括：

- `post_id`
- `view_date`
- `view_count`
- `unique_visitor_count`

该表是生成阅读趋势报表的重要数据来源。

## 8. 指标口径定义

### 8.1 文章总数

文章总数指系统中未被物理删除的文章数量。  
若报表强调“已发布文章总数”，则只统计 `status = published` 的记录。

### 8.2 阅读量

阅读量指文章详情页被访问的次数。  
同一用户多次访问会累计为多次浏览，因此阅读量不等于独立访客数。

### 8.3 独立访客数

独立访客数以自然日为单位统计，通常根据设备标识、Cookie 或登录用户 ID 去重。  
若存在匿名访问，则以客户端访客标识作为去重依据。

### 8.4 评论率

评论率定义为：

`评论率 = 评论数 / 阅读量`

该指标用于衡量文章互动效果。

### 8.5 发布频率

发布频率是指在指定时间范围内发布文章的数量，常按日、周、月统计。  
发布频率过低通常说明内容供给不足，过高则可能影响内容质量。

## 9. 典型查询场景

博客系统通常需要支持以下分析查询：

### 9.1 最近 30 天发文趋势

目标：统计每天发布了多少篇文章。  
常用字段：`posts.published_at`、`posts.status`。

### 9.2 各分类文章数量

目标：按分类统计文章数量并排序。  
常用字段：`posts.category_id`、`categories.name`。

### 9.3 作者发文排行

目标：统计每位作者发布的文章数。  
常用字段：`posts.author_id`、`users.display_name`。

### 9.4 热门文章排行

目标：根据阅读量统计最受欢迎的文章。  
常用表：`posts`、`post_view_daily`。

### 9.5 评论最活跃文章

目标：统计评论数最多的文章。  
常用表：`posts`、`comments`。

### 9.6 标签覆盖率分析

目标：判断某些标签是否被过度使用，或是否有重要主题缺少标签。  
常用表：`tags`、`post_tags`、`posts`。

## 10. 推荐与搜索说明

### 10.1 搜索策略

博客系统搜索通常基于文章标题、摘要、正文、标签名进行匹配。  
搜索排序可考虑以下因素：

- 标题命中权重最高
- 标签命中权重次之
- 摘要命中权重高于正文
- 已发布且近期更新内容可获得轻微加权

### 10.2 推荐策略

推荐策略包括：

- 同分类推荐
- 同标签推荐
- 热门文章推荐
- 最近更新推荐

若系统启用“相关推荐”，优先根据标签重合度与分类一致性进行排序。

## 11. 后台管理规范

### 11.1 命名规范

文章标题应简洁准确，不建议超过 40 个汉字。  
标签名称建议控制在 12 个汉字以内。  
分类名称应保持稳定，避免频繁变更。

### 11.2 数据维护规范

管理员应定期检查以下事项：

- 长期未审核评论
- 长期未发布的草稿
- 分类下无文章的空分类
- 长期未使用的标签
- 发布时间异常或未来时间错误的数据

### 11.3 附件管理规范

上传附件时，应校验文件类型、大小和来源。  
图片建议压缩后上传，单文件默认不超过 10 MB。  
敏感附件应限制下载权限。

## 12. 风险控制与审计

博客系统在生产环境中需要重点关注以下风险：

### 12.1 内容合规风险

需防止文章中出现违规信息、侵权内容和敏感表达。  
可通过敏感词库、审核流程和人工复核降低风险。

### 12.2 评论滥用风险

需防止垃圾评论、恶意灌水和人身攻击。  
建议启用频率限制、敏感词过滤和审核机制。

### 12.3 数据丢失风险

应对文章正文、附件、配置和统计数据执行定期备份。  
生产环境建议每日全量备份、每小时增量备份。

### 12.4 权限误用风险

需要记录关键操作审计日志，包括：

- 文章发布
- 文章下线
- 评论审核
- 用户角色变更
- 系统配置修改

## 16. 结论

博客系统既是内容生产平台，也是运营分析平台。  
在知识库测试场景中，本说明书可用于验证以下能力：

- 是否能正确回答字段、状态、角色、流程类问题
- 是否能返回准确引用片段
- 是否能在文档问答与数据库分析之间切换
- 是否能在联合问题中同时使用知识库与数据库工具

若用于 DataMind 的知识库测试，建议优先提问角色权限、文章状态、评论流程、指标口径和典型查询场景，这几类问题最容易验证检索质量与引用效果。
